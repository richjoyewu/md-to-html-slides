# md-to-html-slides 产品介绍讲稿

## 5 分钟演讲稿

大家好，我想先讲一个很常见、但长期没有被真正解决的问题。

今天大多数人并不缺内容。我们有文档，有笔记，有方案，有课程材料，也有大量已经写出来的文字。但是，文字写出来，不等于能讲清楚。很多时候，真正卡住我们的，不是没有内容，而是不知道怎么把这些内容变成适合表达、适合演讲、适合被别人快速理解的形式。

所以我们现在做的这个产品，目标并不是做一个新的 PPT 工具，也不是再做一个 Markdown 转页面的工具。我们真正想做的是，把文本语言，转成适合演讲的视觉语言，帮助用户更清晰、更简单地表达。

这件事为什么重要？因为文本语言和演讲语言，本来就是两套不同的系统。文本更适合完整、严谨、连续地展开；但演讲不是这样。演讲需要节奏，需要重点，需要视觉重心，需要让观众在很短时间里抓住主线、记住结论。所以很多人会发现，自己文档写得不错，但一到要做介绍、做分享、做 pitch、做汇报，表达效率就突然下降了。

我们认为，这里面缺的不是一个模板库，而是一层“表达转译能力”。也就是说，系统要先理解你真正想表达什么，再判断哪些内容应该上屏，哪些内容应该留给讲解；哪些地方应该拆开，哪些地方应该合并；最后把原来的文本，转成更适合讲述和展示的视觉表达。

这也是为什么我们现在虽然是从 Markdown 和 HTML 切入，但产品本身不应该被定义成一个 HTML 工具，或者一个 PPT 替代品。HTML 只是当前最适合落地和验证的输出方式。我们今天先从 deck 形态开始，因为它最适合承载“讲一件事、解释一条主线”的表达场景。但长期来看，我们希望这个产品服务的是更广义的表达，不局限于 PPT，也不局限于单一页面格式。

如果从用户价值来看，这个产品解决的是三件事。第一，降低表达门槛。用户不用先学会怎么做页，先把内容写出来就行。第二，提高表达清晰度。系统帮助用户把复杂内容整理成更容易讲清楚的结构。第三，提高表达效率。过去需要来回改文稿、改页面、改顺序的过程，可以被明显压缩。

如果从业务角度来看，这个方向也不只是面向某一个小场景。它可以服务产品介绍、融资 pitch、课程内容、内部汇报、方案讲解、briefing，甚至未来更多以“讲清楚”为核心的沟通场景。只要用户面对的问题是“我有内容，但我不知道怎么更好地表达”，这个产品就有价值。

所以对我们来说，当前做的不是一个格式工具，而是一种新的表达层。输入是文本内容，输出是更适合讲述、理解和传播的视觉语言。当前我们先把 deck 做好，但 deck 不是边界，它只是起点。

如果用一句话总结，这个产品想做的事情就是：把文本语言转成适合演讲的视觉语言，帮助用户更清晰、更简单地表达。

## 什么是 deck

在这个产品里，`deck` 可以理解成“当前默认的演讲型展示形态”。

它的核心意思不是 PPT 文件，而是：

- 围绕一条主线展开
- 由多页或多个展示单元组成
- 适合按顺序讲给别人听
- 每一页都服务于整体表达节奏

所以，`deck` 更接近“成套演示内容”这个概念，而不等于 PowerPoint。

你可以把它理解成：

- 一份用于讲解的线性表达结构
- 一组有顺序的视觉化表达单元
- 当前产品最适合落地的默认模式

这也是为什么我们会说：

- 当前产品是 `deck`-first
- 但产品目标不应被永久限制在 `deck`
- 更不应该被误解成“只做 PPT”

简单说：

- `PPT` 是一种文件或软件形态
- `deck` 是一种表达形态

你这个产品当前先落在 `deck`，但长期目标是更大的：
不是生成某一种文件，而是帮助用户完成从文本语言到演讲视觉语言的转译。
