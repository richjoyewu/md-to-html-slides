# Freeform HTML Art Mode 决策文档

## 文档目的

这份文档用于回答一个当前阶段非常关键的问题：

> `md-to-html-slides` 是否应该支持类似 Claude 一次性生成的高审美、强定制 HTML 成品模式？

它不讨论局部样式微调。
它讨论的是产品是否需要一条新的生成模式，以及这条模式会不会破坏现有系统的稳定性。

---

## 要回答的问题

当前项目的主链路是：

`Markdown -> Plan -> Clarification -> Outline Confirmation -> Expand -> HTML`

这条链路的核心目标是：

- 可确认
- 可复用
- 可维护
- 可回归测试
- 渲染确定性强

而用户正在追求的另一类效果是：

- 一次性惊艳
- case-specific 布局
- 更多颜色、图标、装饰和图形
- 为当前这一份内容量身定制的 HTML 成品

这两类目标天然存在张力。

因此当前真正的问题不是“能不能继续调 theme”，而是：

1. 是否应该新增 `Freeform HTML Art Mode`
2. 如果新增，它会不会拖垮现有系统稳定性
3. 如果不新增，是否还应该继续用系统模式去逼近一次性成品效果

---

## 结论

### 结论一：有必要评估 `Freeform HTML Art Mode`

有必要。

原因不是因为当前系统完全错误，而是因为用户想要的两类价值并不相同：

- `System Mode`
  - 更稳定
  - 更可测试
  - 更适合长期产品化
- `Art Mode`
  - 更自由
  - 更容易一次性惊艳
  - 更接近 Claude 这类“直接给成品”的体验

如果长期只保留 `System Mode`，团队会持续拿“通用系统输出”去对比“一次性定制成品”，从而反复陷入错误期待。

### 结论二：如果把 `Art Mode` 直接混进主链路，会明显增加系统不稳定性

会。

而且风险不小。

### 结论三：如果把 `Art Mode` 明确隔离成一条独立模式，它不必然破坏系统稳定性

不会必然破坏。

前提是：

- 不把它混进当前 deterministic renderer
- 不让它污染主链路 contract
- 不用它替代当前 `Plan -> Expand -> Render` 主干

也就是说：

> `Art Mode` 的风险，不在于它存在，而在于它是否被错误地塞进主系统。

---

## 对“会不会导致系统不稳定”的直接回答

短答案：

> 会，如果把它当成现有主链路的一部分硬塞进去。  
> 不一定会，如果把它作为一条明确隔离的可选模式。

更具体地说：

### 会导致不稳定的情况

以下做法会明显增加不稳定性：

- 让 LLM 直接输出最终 HTML，同时仍假装主链路是 deterministic
- 把一次性自由生成和现有 theme renderer 混在一起
- 让 `Art Mode` 复用当前所有质量评测，却不承认它的产物本质不同
- 把“高审美 case-specific 输出”当成对通用系统 renderer 的直接要求

这会导致：

- contract 边界模糊
- 回归测试失真
- 生成结果波动更大
- CLI / Studio / server 的行为更难统一
- 维护成本快速上升

### 不一定导致不稳定的情况

如果采用以下方式，风险可控：

- 明确把 `Art Mode` 作为新的输出模式，而不是主链路升级
- 让它拥有独立的 prompt、独立的质量标准、独立的使用预期
- 保留 `System Mode` 作为默认稳定模式
- 在产品和文档中明确两种模式的差异

这时系统可以保持：

- 主链路继续稳定
- 高自由模式单独试验
- 用户预期更清晰

---

## 两种模式的本质区别

### 1. System Mode

目标：

- 把内容稳定转成展示结构
- 强调 planning、confirmation、rewriting、deterministic rendering

特点：

- contract 清晰
- renderer 可控
- 质量容易评估
- 更适合产品化和长期维护

代价：

- 审美上更保守
- 很难像一次性作品那样惊艳
- 更难直接达到 Claude 式成品感

### 2. Freeform HTML Art Mode

目标：

- 让模型为当前内容直接产出更完整、更定制、更像设计稿的 HTML 成品

特点：

- 自由度高
- 更容易出现强视觉结果
- 更适合做单次展示、demo、视觉样板

代价：

- 结果波动更大
- 可维护性更差
- 回归测试更难
- 对“为什么这次好看、下次不好看”的解释性较弱

---

## 为什么当前阶段不建议立刻实现

虽然方向上有必要，但当前阶段不建议马上做。

原因如下：

### 1. 主链路还在收敛期

当前项目仍在继续收敛这些问题：

- 主链路 contract
- semantic visual block 设计
- renderer 表达能力
- Studio 体验
- 用户对“什么叫好结果”的判断标准

在主链路尚未完全收敛时再新增一条自由模式，容易让判断更加混乱。

### 2. 当前团队还没有稳定的评审标准

目前项目已经有：

- 结构质量判断
- planning / expansion 回归

但还没有一套足够稳定的“高审美成品评审标准”。

在这种情况下引入 `Art Mode`，很容易变成：

- 产出看起来更花
- 但团队无法稳定判断它到底更好还是更差

### 3. 容易掩盖真正的系统问题

如果过早引入 `Art Mode`，团队可能会用它来掩盖当前系统模式的问题，例如：

- semantic contract 还不够强
- compare / metrics / hero 的结构还不够稳
- visual density 还不够好

这会让系统模式长期处于“半成品但没人继续修”的状态。

---

## 推荐决策

### 当前阶段决策

当前阶段：

> 不立即实现 `Freeform HTML Art Mode`。

当前应该做的是：

1. 继续明确 `System Mode` 的边界和价值
2. 统一主链路输出心智
3. 明确“高审美成品”到底是产品长期目标，还是辅助模式

### 下一阶段可接受的动作

下一阶段可以做的不是直接开发，而是：

1. 写清楚两种模式的定位
2. 写清楚评测标准差异
3. 设计模式切换边界
4. 定义最小实验范围

也就是说：

> 先做模式分叉设计，再做模式分叉实现。

---

## 如果未来要做，正确做法是什么

### 产品层

明确区分：

- `System Mode`
- `Art Mode`

并让用户知道：

- 一个更稳定
- 一个更自由

### 架构层

不要让 `Art Mode` 污染当前 deterministic renderer。

推荐结构：

```text
Markdown
  -> System Mode
     -> Plan -> Expand -> Semantic Blocks -> Theme Renderer -> HTML

Markdown
  -> Art Mode
     -> Creative Prompt / Layout Draft / Freeform HTML Generation -> HTML
```

两条模式可以共享：

- 输入清洗
- profile 信息
- 一部分内容理解结果

但不应该强共享：

- renderer contract
- visual QA 标准
- deterministic guarantee

### 评测层

`System Mode` 评测：

- 结构正确性
- 页序合理性
- 改写质量
- 可确认性
- 稳定性

`Art Mode` 评测：

- 审美完成度
- 信息层级
- case-specific 贴合度
- 视觉惊艳度

两者不应使用完全相同的 eval。

---

## 风险清单

如果未来实现 `Art Mode`，需要提前接受这些风险：

1. 输出结果波动更大
2. 更难回归测试
3. 更难解释失败原因
4. HTML 质量更依赖 prompt 和案例
5. 可能出现“这次惊艳、下次失控”的体验波动

如果团队不能接受这些风险，就不应该做。

---

## 当前判断标准

当前阶段判断是否要推进 `Art Mode`，建议只看这三个问题：

1. 用户真正想买单的是“稳定生成系统”，还是“直接出视觉成品”？
2. 团队是否愿意明确维护两种不同模式？
3. 当前主链路是否已经足够清晰，不会被新模式带乱？

只有当这三个问题都得到清晰答案后，才应该进入实现阶段。

---

## 当前最终结论

`Freeform HTML Art Mode` 是一个合理方向，但不是当前阶段应立即落地的功能。

对“它会不会导致系统不稳定”的最终判断是：

> 如果把它混进现有主链路，会。  
> 如果把它隔离成独立模式，不一定会。  
> 当前阶段最合理的动作不是开发它，而是先完成模式分叉判断。
