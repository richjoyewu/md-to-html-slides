# 面试口播稿与 Q&A

## 一、项目介绍口播稿（约 2-3 分钟）

### 开场：一句话说清楚项目是什么

> 我做了一个基于 LLM Agent 的演示稿生成产品。用户只需要输入讲稿、笔记、或者大纲这类原始内容，系统会自动理解内容、规划演示结构、改写成适合上屏的表达，最终生成可以直接分享的网页演示稿。

### 为什么做这个：问题定义

> 做这个项目的起点是我观察到一个普遍痛点：很多人有内容，但不会做演示稿。他们的输入往往是讲稿、散乱笔记、或者课程大纲，不是演示结构。他们不知道一页该讲什么、怎么拆页、怎么把说明文改成演示语言。
>
> 市面上的工具要么只做 Markdown 渲染，完全不理解内容；要么是重型 PPT 编辑器，学习成本太高。中间缺一个"懂内容"的轻量化方案。
>
> 所以我定义这个产品的核心问题不是"渲染"，而是**把原始内容自动转化成适合展示的结构化演示稿**。

### 怎么做的：架构设计

> 我设计了一条 6 阶段的 Agentic Pipeline：
>
> 第一步，**结构分析**，系统先分析输入的结构特征——信息密度、粗糙度、是否像演示文稿，但这一步不做语义判断，只提供事实。
>
> 第二步，**主动澄清**，如果 LLM 对规划没有足够把握，比如 planning confidence 低于阈值或者存在明确的 uncertainties，系统会向用户追问 1-2 个关键问题——比如"给谁看"或"希望几页"。但核心原则是：能自动做就自动做，不确定才问，最多问两个。
>
> 第三步，**LLM 规划大纲**，这是整个系统的核心。我把 LLM 定义为"主脑"，它负责理解内容在讲什么、判断该拆成几页、每页保留什么核心点。LLM 会输出结构化的大纲，包括每页标题、预览要点、详细要点，以及它自己的置信度和不确定性。
>
> 第四步，**自检纠偏**，系统会对大纲做质量检查——标题是否过长、某页是否过满或过薄、相邻页是否重复，对明显问题做自动修正。
>
> 第五步，**演示语言改写**，用户确认大纲后，系统把内容改写成适合上屏的短句——去掉讲稿腔、bullet 平行化、结论先行。
>
> 第六步，**确定性渲染**，输出最终 HTML，这一步不用 LLM，保证渲染结果稳定可控。

### 设计原则

> 整体架构我遵循一个核心原则：**LLM 是主脑，系统是轨道**。
>
> LLM 负责内容理解、结构规划、判断是否要问用户。系统负责兜底、缓存、超时处理、状态机和渲染。规则只做三件事：结构提取、可靠性兜底、输出约束。不让规则替 LLM 做主语义判断。
>
> LLM 挂了或超时怎么办？系统有完整的 fallback 机制，会用规则生成一版启发式大纲，保证用户永远不会卡在空白页。

### 结果

> 最终产品实现了完整的端到端闭环：输入内容 → 大纲确认 → 预览导出。我还做了一个 Studio 在线编辑器，支持实时预览和主题切换。
>
> 同时我建立了一套质量评估体系，包括 7 个自动评分维度和 5 个 Agent 评估指标，北极星指标是"用户第一次生成后的改动量"——改动越少，产品越好。

### 收尾：关于 AI 协作开发

> 工程实现上，我借助 AI 编程工具协作完成。这个过程本身也让我对 LLM 能力边界有了一手体感——哪些环节适合交给模型做主判断，哪些必须用规则兜底，这些认知直接影响了我的产品设计决策。

## 二、高频面试问题与回答

### Q1：为什么选择单 Agent 而不是多 Agent 架构？

> 我的判断是：当前阶段单 Agent 还没做强，引入多 Agent 会显著增加系统复杂度，但不会提升产品质量。
>
> 核心瓶颈不是"协同"，而是"主脑质量"——大纲规划得好不好，直接决定了最终结果的上限。如果 Planner 本身的理解能力不够，拆成多个 Agent 并不会让它变聪明，只会让调试变难。
>
> 如果后续要扩展，我会优先考虑"主 Agent + Reviewer Agent"的模式——让一个 Agent 规划，另一个 Agent 审查，而不是一开始就拆很多执行 Agent。

### Q2：Human-in-the-Loop 是怎么设计的？什么时候问用户，什么时候不问？

> 我的原则是四个字：**能做就做，不确定才问**。
>
> 具体来说，触发澄清的信号不来自规则预判，而来自 LLM 自己的输出。Planner 会输出两个关键字段：planning_confidence（规划置信度）和 uncertainties（不确定点列表）。
>
> 只有两种情况会触发澄清：一是 confidence 低于 0.65，二是 uncertainties 中包含关键信息缺失（比如不知道受众是谁）。
>
> 触发后最多问 1-2 个问题，问完必须能继续规划，不能停住。这样设计是为了避免两个极端：什么都自己猜（结果不可控），或者不停追问（体验很差）。

### Q3：如果 LLM 返回的结果质量很差怎么办？

> 我设计了三层保障：
>
> **第一层，Normalize**：LLM 的 JSON 输出经常不稳定——字段缺失、类型不对、格式异常。Normalize 层会做结构清洗，把不稳定的 LLM 输出修正成系统可以继续处理的标准格式。
>
> **第二层，Polisher**：对大纲做质量自检——标题过长就压缩，某页过满就拆分，过薄就合并，相邻页重复就去重，summary 不在最后就后置。
>
> **第三层，Fallback**：如果 LLM 完全不可用（超时、报错、返回垃圾），系统会用规则生成一版启发式大纲。质量不如 LLM 版本，但保证用户不会卡住。这是可用性的底线。

### Q4：你怎么评估这个 Agent 产品做得好不好？

> 我定义了两个层面的指标。
>
> **Agent 层面**：
> - outline_accept_rate：用户是否接受大纲，不需要打回重做
> - clarification_quality：问题是否少而准，问了之后 planning 是否真的改善
> - rewrite_quality：输出是否像演示稿，而不是原文复读
> - self_correction_rate：Polisher 是否减少了明显的结构问题
> - fallback_hit_rate：fallback 是否过于频繁（频繁说明 LLM 环节不稳定）
>
> **产品层面**，我用了一个北极星指标：**用户第一次生成后需要手改多少**。改动越少，产品越好。这个指标把内容理解、结构规划、改写质量全部串在一起了。
>
> 工程上，我还建了自动化评分脚本，覆盖 segmentation、title_length、density、duplication 等 7 个维度，每次改动后可以跑回归检查，防止退化。

### Q5：为什么 LLM 是主脑，规则不行吗？

> 规则的问题是：它只能处理你预见到的情况。
>
> 用户的输入千差万别——有人给你一篇完整论文，有人给你三行笔记，有人给你一段会议纪要。如果用规则做内容理解，你需要枚举所有输入类型，然后为每种类型写分支逻辑。这条路很快就会变成无底洞，而且每加一条规则都可能影响已有逻辑。
>
> LLM 的优势是：它天然能理解"这段话在讲什么"、"应该怎么拆"、"哪些是重点"。所以我让 LLM 做主语义判断，规则只做结构提取和可靠性兜底。
>
> 但 LLM 也有弱点——输出不稳定、可能超时、可能幻觉。所以系统要搭好轨道：Normalize 清洗输出、Polisher 自检纠偏、Fallback 保底。这就是"LLM 主脑 + 系统轨道"的含义。

### Q6：你对 Agent 和 Chatbot 的区别怎么理解？

> 最核心的区别是：**Chatbot 响应用户指令，Agent 主动规划和执行任务**。
>
> 在我的产品里，用户不需要一轮一轮地告诉系统"先帮我分析一下"、"然后帮我拆页"、"再帮我改写"。用户只需要输入内容，系统自己决定下一步做什么——需要分析就分析，需要问就问，需要改写就改写。
>
> 具体来说，Agent 和 Chatbot 有三个关键差异：
> 1. **Agent 会规划**：它有一个 Planner，会把大任务拆成小步骤
> 2. **Agent 会自检**：Polisher 会审查自己的输出，发现问题会修正
> 3. **Agent 知道自己不知道什么**：它会输出 uncertainty，只在关键信息缺失时问用户，而不是每轮都等指令

### Q7：你在这个项目中用 AI 工具协作开发，怎么看 AI 辅助编程？

> 我的体验是：AI 编程工具在"执行明确定义的任务"时效率非常高，但在"做产品判断"时仍然需要人来主导。
>
> 比如，Pipeline 该分几个阶段、每个阶段的职责边界在哪里、什么时候该问用户什么时候不该问、评估指标怎么定义——这些产品设计决策是我做的。具体的代码实现、接口联调、测试脚本，AI 工具可以高效完成。
>
> 这个经历也直接影响了我的产品设计。因为我亲手体验了 LLM 擅长什么、不擅长什么，所以我知道哪些环节可以放心交给模型，哪些必须用规则兜底。这种一手体感，是只做 PRD 的 PM 很难获得的。

### Q8：如果让你继续迭代这个产品，下一步做什么？

> 我的 P1 规划聚焦在三件事：
>
> **第一，让 Planner 真正成为主脑**。当前 analysis 模块还有一些规则在替 LLM 做分类判断，我要把这些逐步收缩，让 LLM 直接输出 content_intent、audience_guess 等语义信息，减少规则对主路径的干预。
>
> **第二，让 Clarification 更智能**。现在的澄清机制还有一些硬编码的触发条件，理想状态是完全依赖 LLM 的 uncertainty 来驱动——LLM 不确定什么，就问什么；确定的就直接做。
>
> **第三，强化评估体系**。增加 outline_acceptability_score 和 manual_edit_cost_estimate 两个新指标，让质量评估更贴近真实用户体验。
>
> 明确不做的事：不做多 Agent 拆分、不做 PPT 导出、不做复杂记忆系统。当前阶段把单 Agent 做强，比堆功能更重要。

### Q9：你怎么定义一个好的 AI Agent 产品？

> 三个标准：
>
> **第一，会做主判断**。不是等用户一步步指令，而是自己能判断下一步做什么。在我的产品里，Planner 就是做这个的。
>
> **第二，知道自己不知道什么**。一个好的 Agent 不会什么都猜，也不会不停地问。它会评估自己的置信度，只在关键信息缺失时寻求人类输入。
>
> **第三，出了问题能兜住**。LLM 天然不稳定，所以 Agent 产品必须有 fallback、有自检、有降级策略。用户感知到的应该是稳定可靠的产品，而不是一个随机性很强的 demo。
>
> 一句话总结：**好的 Agent 产品 = 会做主 + 知道边界 + 兜得住底**。

### Q10：这个项目有用户吗？数据怎么样？

> 这是一个从 0 到 1 的个人项目，目前没有外部用户和线上数据。但我做了两件事来弥补：
>
> 第一，我建立了 fixtures 样本库，覆盖了 course、report、pitch、story 四类内容和 clean、rough、extreme 三个质量层级，用自动化测试做回归验证。
>
> 第二，我定义了完整的评估指标体系。虽然没有真实用户数据，但指标定义本身体现了我对"产品做得好不好"的思考框架。如果产品上线，这套指标可以直接接入埋点系统，驱动数据化迭代。
>
> 作为 AI PM，我认为在早期阶段，能定义清楚"什么是好"，比有一堆模糊的用户数据更重要。
