# 内部产品原则

## 文档目的

这份文档用于定义 `md-to-html-slides` 的内部产品原则。

它不是对外宣传文案，也不是营销介绍。
它的用途是帮助我们判断：
- 这个产品真正要解决什么问题
- 什么属于 MVP 主链路
- 什么应该交给 agent 决策
- 什么必须由系统保证确定性
- 哪些功能即使看起来很吸引人，也应该延后

## 核心问题

这个产品要解决的，不是狭义上的 `Markdown 转 HTML`。

真正的问题是：

> 用户有内容，但不知道如何稳定地把这些内容变成一份结构清楚、适合展示、可以直接发布的展示形态。

这个问题通常表现为四种失败：
- 用户不知道如何把内容拆成适合展示的单元
- 用户不知道如何把文稿语言压缩成演示语言
- 用户不知道哪些内容应该上屏，哪些内容应该留给讲解
- 用户不知道如何把结构进一步变成高质量的 HTML 演示稿

任何不能改善这条核心转化链路的功能，都不是当前阶段的主问题。

## 产品定义

这个产品应该定义成：

> 一个用 agent 把 Markdown 或文稿整理成可确认展示结构，再生成适合讲解的 HTML 展示形态的系统。

这意味着它不应被定义成：
- 一个 Markdown 编辑器
- 一个主题库
- 一个 HTML 导出器
- 一个普通的 slide 渲染器

这些都只是支撑层。
真正的核心价值，是把“原始内容”转成“可展示结构”。

在当前实现阶段，产品仍然以 `deck` 模式为主。
但产品定义本身不应被永久限制在 deck 上。
后续应该逐步支持多种展示模式，例如：

- `deck`
- `roadmap`
- `briefing`
- `storyflow`

## 当前主链路

当前项目最需要守住的主链路是：

`Markdown -> Plan -> Clarification -> Outline Confirmation -> Expand -> HTML`

这里的关键不是 HTML 渲染，而是把文稿逻辑转成展示逻辑。

因此系统的职责是：

1. 理解输入是否已经接近 slide，还是更像文稿
2. 规划每一页讲什么
3. 在不确定时用最少的问题补足信息
4. 把确认后的大纲改写成适合上屏的表达
5. 用确定性的渲染层输出结果

这条链路应该始终优先于外围能力和附属功能。

## 原则一：用户提供内容，不提供结构

用户不应该被要求理解系统内部的数据形态。

用户不应该被迫区分：
- draft markdown
- slide markdown
- planner 输入
- renderer 输入

对用户来说，只有一件事：
- 提供内容

系统应该自己判断，这份内容是否需要：
- 补充信息
- 拆页
- 改写
- 扩展
- 润色

这意味着：
- UI 不要暴露太多技术抽象
- 不要过早要求用户学习一套“课件专用语法”
- 用户界面尽量使用简单直接的文案，例如：`输入内容`、`生成展示大纲`、`确认结构`

## 原则二：Agentic 的价值在于更好的决策，不在于更大的自由度

agent 的价值，只体现在它是否提高了决策质量。

在这个产品里，agentic 行为应该包括：
- 判断输入已经接近 slide，还是仍然更像文稿
- 判断是否需要先向用户提问
- 判断这份内容更适合哪种展示模式
- 判断大致应该拆成多少页或多少段
- 判断每一个展示单元的核心点是什么
- 判断哪一页或哪一段信息过载，需要拆开
- 判断哪些长句需要改写成适合上屏的表达

agentic 不应该意味着：
- 让模型自由生成最终 HTML
- 让模型随意重构整个产品流程
- 让不确定性侵入本该稳定的系统边界

这意味着：
- LLM 负责 planning 和 rewriting
- 渲染层必须保持确定性
- 输出结构必须有清晰 schema

## 原则三：只有在确实影响 planning 结果时，才向用户提问

系统不应该频繁追问用户。

默认规则应该是：
- 如果系统可以较可靠地规划，就直接规划
- 如果缺少的信息会明显影响大纲质量，才提问
- 一旦提问，只问最少的问题

好的 clarification 问题包括：
- 这份内容给谁看
- 这是课程、汇报还是宣传
- 你大概希望做几页或几个阶段

不好的 clarification 行为包括：
- 连续追问很多轮
- 问一些不会改变结果的问题
- 因为实现不够强就把问题推回给用户

这意味着：
- clarification 应该是一个 gate，而不是聊天模式
- 默认最多只问 1 到 2 个关键问题

## 原则四：主链路必须保持收敛

MVP 的主链路应该始终是：
1. 输入内容
2. 生成大纲
3. 确认大纲
4. 扩展展示内容
5. 预览展示形态
6. 导出 HTML

任何超出这条链路的功能，都是次要的。

这意味着，以下内容在 MVP 阶段都不是核心：
- 文件上传系统
- 在线协作
- 大量高级主题
- 动画配置面板
- 大型图标系统
- 复杂格式 DSL

这意味着：
- 在主链路不稳定之前，不要扩大产品边界
- 在 planning 还不强之前，不要把精力优先放在外围界面增强上

当前实现仍应优先把 `deck` 主链路做好。
但从产品定义上，它应该被看作“展示形态系统”，而不是“PPT 替代品”。

## 原则五：可靠性比惊艳更重要

基于 LLM 的产品，只有在“可预测”时才真正有用。

因此系统必须内建这些能力：
- 明确状态流转
- clarification gate
- cache
- fallback
- timeout 边界
- 严格 schema
- 用户确认点

一个偶尔很惊艳但经常失败的结果，不如一个稳定可用的普通结果。

这意味着：
- 渲染必须 deterministic
- fallback 不是可选项，而是产品能力
- cache 不只是实现细节，而是用户体验的一部分

## 原则六：不确定性交给 agent，稳定性交给系统

这个项目需要把“不确定决策”和“确定执行”分开。

推荐的分层是：
- agent 层：
  - analyzer
  - planner
  - clarification
  - expander
  - 未来的 polisher
- system 层：
  - cache
  - timeout
  - request lifecycle
  - state machine
  - HTML 渲染
  - preview orchestration

这意味着：
- 不要让 LLM 的自由输出直接控制渲染行为
- 不要把 agent 和 renderer 合并成一步

## 当前模块边界

当前最合理的模块边界是：

- `Analyzer`
  - 负责结构分析
  - 不负责最终语义判断
- `Planner`
  - 负责页序、标题、展示单元聚焦
- `Clarification Gate`
  - 只在确实影响 planning 质量时提问
- `Expander`
  - 负责把大纲改写成上屏表达
- `Polisher`
  - 负责有限度的结构纠偏
- `Renderer`
  - 负责确定性 HTML 输出

这些模块的关系应该保持清晰，不要把 provider、渲染和内容判断重新耦合在一起。

## 原则七：大纲质量决定后面一切

大纲是系统里杠杆最高的产物。

如果大纲错了：
- expand 的质量会直接崩掉
- preview 会误导用户
- 再好的模板也救不回来

所以最高优先级的智能工作应该放在：
- 更好的 planning
- 更好的 clarification
- 更强的展示单元聚焦能力

这意味着：
- planning 的优先级高于新主题
- planning 的优先级高于图片支持（至少在早期）
- outline confirmation 不是可选增强，而是必要产品环节

## 原则八：UI 只暴露当前阶段真正需要做的决策

每一步页面都只应该突出当前阶段的动作。

例如：
- 在生成大纲之前，用户只需要关注输入
- 大纲生成之后，用户只需要关注每页结构是否合理
- 大纲确认之后，用户只需要关注预览、模板和导出

不好的 UI 行为包括：
- 过早展示后续阶段的大容器
- 让辅助信息和主动作争夺注意力
- 用户只需要一个动作，却给他一个 dashboard

这意味着：
- 优先使用分步骤视图
- 保持界面低噪音
- 删除不能提高决策质量的装饰性界面

## 原则九：产品应该隐藏内部复杂度

系统内部可以很复杂，例如：
- analyzer
- planner
- clarification gate
- expander
- polisher
- fallback mode
- cache mode
- state machine

但用户不应该被迫理解这些东西。

用户理想中的感受应该只是：
- 我给了内容
- 系统理解了我要表达什么
- 只有在必要时才问我问题
- 它给了我一个能确认、能发布的结果

这意味着：
- 主界面不要暴露太多内部模块名
- 除非未来战略明确，否则不要把产品做成“AI 工作流控制台”

## 原则十：产品要增强人的判断，不是盲目替代人的判断

这个产品不应该假设模型永远正确。

## 文档使用原则

如果未来文档继续收敛，这份文档应继续承担：

- 产品问题定义
- 主链路定义
- agent 与系统的边界原则
- 什么该做、什么不该做

更具体的实现优先级和工程决策，应放在 `next-technical-strategy.zh-CN.md`。

系统的目标，不是完全把人拿掉。
系统的目标，是放大人的判断力。

这意味着：
- 系统负责提出结构建议
- 用户负责确认关键结构节点
- 系统负责承担重的转化工作
- 用户在高价值节点保留控制权

这意味着：
- 大纲确认必须先于 expand
- 允许用户回退修改大纲

## 原则十一：展示模式是产品决策，不是主题技巧

系统后续应该明确区分：

- `presentation mode`
- `theme`
- `template`

其中：

- presentation mode 决定结构和导航
- theme 决定视觉语言
- template 决定在某个 mode + theme 下的受控默认样式

这意味着：

- roadmap 不应该被硬塞进不合理的翻页 slide 边界
- theme 不应该只靠 CSS 假装成另一种展示模式
- planner 后续应该具备推荐或选择展示模式的能力

这意味着：

- 渲染契约后续要逐步支持 mode-aware
- mode 数量要少而明确
- 把 `deck` 视为当前默认模式，而不是永久产品边界
- 未来的润色层应该尽量可见，而不是完全暗箱改写

## 当前 MVP 优先级

P0：
- 稳定的 plan 生成
- 稳定的 expand 生成
- 清晰的状态机
- clarification gate 闭环
- cache 和 fallback 覆盖主链路

P1：
- 更强的 planner 质量
- 更精准的 clarification
- 更像演示稿的 expand 改写
- 更真实的流式进度

P2：
- polisher
- 图片支持
- 图标支持
- 更丰富的 block 类型

P3：
- 更多 provider 支持
- 更深的 orchestration
- 更复杂的产品层能力

## 功能决策过滤器

在实现任何功能前，先问这几个问题：
1. 它是否改善了“从原始内容到演示结构”的主链路？
2. 它是否减少了用户在主链路上的心智负担？
3. 它是否提高了系统可靠性，而不只是看起来更厉害？
4. 用户是否能立刻感知到它的价值？
5. 如果暂时不做它，核心产品还能不能成立？

如果前两个问题回答得不强，这个功能大概率就不是当前核心。

## 总结

这个产品最终应该用一个标准来判断：

> 它是否可靠地帮助用户把原始内容变成一个可确认、可发布的演示结构？

如果答案是肯定的，产品方向就是对的。
如果不是，那这个功能即使看起来很高级，也大概率只是外围增强。
